## Section 1: Stats APIs (Bot → Dashboard)

The bot exposes these endpoints for the dashboard to pull data.

**Base URL:** `https://{BOT_SERVER}/api/stats`
**Authentication:** `X-API-Key: {ADMIN_API_KEY}` header

```
GET /api/stats/overview
```

**Response:**

{
  "total_conversations": 45,

  "messages_today": 156

## Section 2: Notification Webhooks (Backend → Bot)

The backend calls these endpoints to trigger customer notifications.

**Base URL:** `https://{BOT_SERVER}/webhooks/chicx`
**Authentication:** `X-CHICX-Secret: {CHICX_API_KEY}` header
**Content-Type:** `application/json`

### 2.1 Send OTP

**When to call:**
- User logs in with OTP (passwordless login)

```
POST /webhooks/chicx/send-otp
```

**Request Body:**
```json
{
  "phone": "9876543210",
  "otp": "123456"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| phone | string | Yes | Customer phone (10 digits) |
| otp | string | Yes | 6-digit OTP generated by backend |

**Response (Success):**
```json
{
  "status": "ok",
  "message": "OTP sent to 9876543210",
  "phone": "9876543210",
  "wa_message_id": "wamid.xxx"
}
```

**Response (Error):**
```json
{
  "status": "error",
  "message": "WhatsApp API error: invalid phone number",
  "phone": "9876543210"
}
```

**Example cURL:**
```bash
curl -X POST https://bot.chicx.in/webhooks/chicx/send-otp \
  -H "Content-Type: application/json" \
  -H "X-CHICX-Secret: your_api_key" \
  -d '{
    "phone": "9876543210",
    "otp": "123456"
  }'
```

**Important Notes:**
- OTP generation and verification happens on CHICX backend
- This bot only DELIVERS the OTP via WhatsApp
- Backend should implement OTP expiry (recommended: 10 minutes)
- Backend should implement rate limiting (recommended: max 5 OTPs per phone per hour)

**API Field → Template Variable Mapping:**

| Template Part | Variable | API Field | Fallback | Required |
|---------------|----------|-----------|----------|----------|
| Body | `{{1}}` | `otp` | — | ✅ Yes |
| Button (Copy Code) | `{{1}}` | `otp` | — | ✅ Yes |

> [!IMPORTANT]
> Both `phone` and `otp` are **required**. The API will return a validation error if either is missing.

### 2.2 Order Status Update

**When to call:** Order status changes (placed, shipped, out for delivery, delivered)

```
POST /webhooks/chicx/order-update
```

**Request Body:**
```json
{
  "phone": "9876543210",
  "order_id": "ORD123456",
  "order_status": "Shipped",
  "tracking_url": "https://shiprocket.com/track/abc",
  "delivery_date": "2025-12-30"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| phone | string | Yes | Customer phone |
| order_id | string | Yes | Order ID |
| order_status | string | Yes | New status |
| tracking_url | string | No | Tracking link |
| delivery_date | string | No | Expected delivery date |

**Recommended Status Values:**
- `Order Placed`
- `Order Confirmed`
- `Shipped`
- `Out for Delivery`
- `Delivered`
- `Cancelled`

**Response:**
```json
{
  "status": "ok",
  "message": "Order update notification sent to 9876543210",
  "order_id": "ORD123456",
  "order_status": "Shipped",
  "wa_message_id": "wamid.xxx"
}
```

**API Field → Template Variable Mapping:**

| Template Part | Variable | API Field | Fallback | Required |
|---------------|----------|-----------|----------|----------|
| Body | `{{1}}` | `order_id` | — | ✅ Yes |
| Body | `{{2}}` | `order_status` | — | ✅ Yes |
| Button (URL) | Dynamic suffix | `tracking_url` | Button not sent | No |

> [!IMPORTANT]
> `phone`, `order_id`, and `order_status` are all **required**.

> [!NOTE]
> **Marketing notifications** (cart reminders, new product announcements, sales) are handled via **AiSensy** platform, not through this bot. See [`BACKEND_AISENSY_INTEGRATION.md`](./BACKEND_AISENSY_INTEGRATION.md) for marketing automation setup.

### 2.3 Order Confirmation Call (COD)

**When to call:** COD order placed, needs verbal confirmation

```
POST /webhooks/chicx/confirm-order
```

**Request Body:**
```json
{
  "phone": "9876543210",
  "order_id": "ORD123456",
  "customer_name": "Priya",
  "items": [
    {"name": "Gold Plated Necklace Set", "qty": 1, "price": 2499},
    {"name": "Silver Earrings", "qty": 1, "price": 799}
  ],
  "total_amount": 3298.00,
  "cod": true,
  "delivery_address": "123 Main Street, Chennai 600001"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| phone | string | Yes | Customer phone |
| order_id | string | Yes | Order ID |
| customer_name | string | No | Customer name |
| items | array | No | Order items list |
| items[].name | string | Yes | Item name |
| items[].qty | int | No | Quantity (default: 1) |
| items[].price | number | No | Item price |
| total_amount | number | Yes | Order total |
| cod | boolean | No | Is COD order (default: false) |
| delivery_address | string | No | Delivery address |

**Response:**
```json
{
  "status": "ok",
  "message": "Confirmation call initiated for order ORD123456",
  "order_id": "ORD123456",
  "call_id": "bolna_call_xyz",
  "phone": "9876543210"
}
```

**What happens after the call:**
The bot will analyze the customer's response and POST the result to your backend at `/api/confirm_order.php` (see Section 3.5).

### 3.5 Confirm Order Callback (Bot → CHICX)

> [!IMPORTANT]
> Your backend **must implement** this endpoint. The bot calls it after a confirmation voice call completes.

```
POST /api/confirm_order.php
Authorization: Bearer {CHICX_API_KEY}
```

**Payload the bot sends:**
```json
{
  "order_id": "ORD123456",
  "confirmed": true,
  "confirmation_method": "voice_call",
  "notes": "Customer confirmed order via voice call"
}
```

| Field | Type | Description |
|-------|------|-------------|
| order_id | string | The order ID you sent in the confirm-order request |
| confirmed | boolean | `true` = confirmed, `false` = rejected or no answer |
| confirmation_method | string | Always `"voice_call"` |
| notes | string | Details: "Customer confirmed...", "Call not answered: missed", "Customer rejected/cancelled...", "Customer response unclear" |

**Your backend should:**
1. Look up the order by `order_id`
2. If `confirmed == true` → proceed with order processing (ship the order)
3. If `confirmed == false` → hold or cancel the order based on `notes`:
   - `"Call not answered: missed"` → Retry later or cancel
   - `"Customer rejected/cancelled..."` → Cancel order immediately
   - `"Customer response unclear"` → Manual review or retry
4. Return `{"status": true}` on success

**Example Response:**
```json
{
  "status": true,
  "message": "Order confirmation processed"
}
```

> [!CRITICAL]
> This endpoint is **REQUIRED** for COD order confirmation flow to work. Without it, confirmation calls will complete but the backend won't know the result.

## Section 5.1: WhatsApp Two-Number Setup

The bot supports two separate WhatsApp numbers following Meta's best practices:

### Message Routing

| Message Type | Category | WhatsApp Number |
|--------------|----------|-----------------|
| Customer chat (bot conversations) | Service | PRIMARY |
| OTP (login, forgot password, purchase) | Authentication | PRIMARY |
| Order status updates | Utility | PRIMARY |
| Cart abandonment reminders | Marketing | MARKETING |
| New product announcements | Marketing | MARKETING |
| Sale/discount notifications | Marketing | MARKETING |

### Why Two Numbers?

1. **Rate Limits**: Marketing messages have stricter limits; won't affect OTP delivery
2. **User Trust**: Customers expect OTP/order updates; marketing feels less intrusive on separate number
3. **Opt-out Management**: Users can block marketing without losing transactional messages
4. **Compliance**: Easier to manage consent for marketing vs transactional
5. **Cost Tracking**: Different rates per category; easier to track costs

### Bot Environment Variables

```env
# PRIMARY (Chatbot/Transactional)
WHATSAPP_PHONE_NUMBER_ID=primary_phone_id
WHATSAPP_ACCESS_TOKEN=primary_access_token
WHATSAPP_APP_SECRET=primary_app_secret

# MARKETING (Optional - falls back to primary if not set)
WHATSAPP_MARKETING_PHONE_NUMBER_ID=marketing_phone_id
WHATSAPP_MARKETING_ACCESS_TOKEN=marketing_access_token
WHATSAPP_MARKETING_APP_SECRET=marketing_app_secret
```

### Fallback Behavior

If marketing credentials are not configured, all messages will be sent via the primary number. This allows gradual migration to two numbers.

## Section 6: Error Handling

All endpoints return errors in this format:

```json
{
  "status": "error",
  "message": "Description of what went wrong",
  "code": "ERROR_CODE"
}
```

**Common Error Codes:**
| Code | HTTP Status | Description |
|------|-------------|-------------|
| UNAUTHORIZED | 401 | Invalid or missing API key |
| NOT_FOUND | 404 | Resource not found |
| VALIDATION_ERROR | 400 | Invalid request payload |
| RATE_LIMITED | 429 | Too many requests |
| INTERNAL_ERROR | 500 | Server error |

### Common Issues & Solutions

#### Issue 1: "Product search returns empty results"
**Cause:** Response format mismatch  
**Solution:** Ensure API returns `"data"` array (not `"products"`) with `"total_records"` field

#### Issue 2: "Order not found for user"
**Cause:** Phone number format mismatch  
**Solution:** Store phone numbers as 10 digits without `+` prefix (e.g., `9876543210`)

#### Issue 3: "Confirmation callback not received"
**Cause:** `/api/confirm_order.php` endpoint not implemented  
**Solution:** Implement the endpoint as documented in Section 3.5

#### Issue 4: "WhatsApp template not found"
**Cause:** Template not approved in Meta Business Manager  
**Solution:** Create and get approval for all templates listed in Section 4

#### Issue 5: "Unauthorized webhook call"
**Cause:** Missing or incorrect `X-CHICX-Secret` header  
**Solution:** Include the header with correct API key in all webhook requests
